## Running the program

This project expects a working `MLIR/LLVM` build with `MLIRConfig.cmake` available. Repository cloning, build and compilation is performed using the following commands:

```bash
git clone git@github.com:RTCupid/MLIR_MyTilingPass.git
cd MLIR_MyTilingPass
cmake -S . -B build -DMLIR_DIR=/path/to/lib/cmake/mlir
cmake --build build
```

Program execution is performed in the following format:

```bash
cd build/bin
./my-tiling-opt <mlir_program> -my-tiling
```

Also this tool contain options for `M`, `N`, `K` dimensions of tiling. Entry `-my-tiling="tile-sizes=%s,%s,%s"`.

## Annotation
A tool based on `mlir-opt` has been developed to perform tiling for matrix multiplication operations in the `Linalg` dialect. The implementation utilizes `TableGen` specifications and a `C++` class derived from `MyTilingPassBase`, which is generated by `TableGen`. The `TilingInterface` was used to identify suitable operation patterns, and `scf::tileUsingSCFForOp` was employed to execute the transformation with tiling.

The tool was successfully tested across a wide range of configurations: matrices with dimensions divisible and not divisible by tile sizes; dynamic shapes; two sequential matrix multiplication operations; rectangular matrices with various `M`, `N`, `K` values; `tensor` and `memref` types; as well as `linalg.matmul` and `linalg.generic` operations. Additionally, correctness was verified in cases where the matrix multiplication operation was originally located inside a loop.

## Introduction
In modern tensor compilers for deep learning, matrix multiplication (`matmul`) is the dominant operation in terms of computational cost. It accounts for the majority of execution time in both training and inference. The efficiency of matmul implementation heavily depends on how memory accesses are organized: without considering the memory hierarchy and target architecture, code generation results in excessive `DRAM` traffic, poor `cache` utilization, and underutilization of computational units.

A naive implementation using nested loops does not reorder data accesses. Each access loads an entire `cache line`, and repeated traversals over the same data cause massive unnecessary transfers between memory levels. Data reuse is minimal, performance becomes bounded by memory bandwidth, and `vector` instructions along with `core‑level` parallelism remain unused.

The proposed `tiling` pass addresses these issues by partitioning the operation into blocks. Tile sizes are selected to match `cache capacity`, maximizing data reuse and reducing `cache misses`. Additionally, the pass automates the transformation, eliminating the need to manually write complex loop nests. Its general design correctly handles dynamic shapes, rectangular matrices, and scenarios where the matmul operation is already nested inside loops.

## Methodology
The tiling pass is implemented using the `MLIR` framework, which provides a flexible intermediate representation (`IR`) and a comprehensive set of transformation utilities. Since matrix multiplication operations are part of the `Linalg` dialect, the optimization pass is specifically designed for this dialect.

In `MLIR`, there are two primary approaches to implementing such transformations.

The first is the pattern-oriented approach, which involves creating a set of classes that inherit from `OpRewritePattern` and applying them through rewriting mechanisms such as `applyPatternsAndFoldGreedily`. This method is convenient when combining multiple related transformations, but it requires explicit management of the order in which patterns are applied.

The second approach is iterative operation traversal—specifically, manually iterating over all operations in the module and checking for the implementation of the `TilingInterface`. This approach provides full control over the transformation process and simplifies debugging, as it does not rely on the nondeterministic traversal inherent to pattern-based rewriting.

The project is built in a `standalone` configuration, which isolates the development of the test pass from the main `MLIR` source tree and simplifies its integration into external tools.

## Tiling Pass Implementation
The first iteration of the tiling pass was implemented using rewrite patterns. This approach allowed leveraging existing mechanisms, such as `applyPatternsAndFoldGreedily`, for iterative application of transformations. The most significant challenges at this stage involved configuring dependencies between files and determining the available set of already implemented functions.

In the pass, the problem of re-applying tiling to already processed operations emerged. To solve this, the iteration count in `applyPatternsAndFoldGreedily` was limited, and a check was added to verify whether an operation belonged to a loop — this highly indicated that the operation had already undergone the transformation. The implemented pass successfully passed most tests. Nevertheless, additional difficulties arose when dealing with matrix multiplication originally located inside a loop.

Consequently, a decision was made to transition to a pass implementation based on operation `walk` and direct calls to the `scf::tileUsingSCFForOp` function. This approach enabled more precise control over the transformation process and simplified the implementation of a single pass over all operations supporting the `TilingInterface`. However, during further experiments, it became clear that for integration with other passes and for using the `GreedyPatternRewriter` mechanism, the pattern-oriented style is more convenient.

In the end, the pass was rewritten using patterns, but with the addition of a simple `tiled` attribute, which was set on operations after a successful tiling transformation. This prevented the transformation from being applied again to already processed operations and also allowed compatibility with `applyPatternsAndFoldGreedily`.

To automatically generate the pass base class, a description was created in `TableGen` language (see [Passes.td](/include/MyTiling/Passes.td)). It defines the pass parameters (`tile sizes`) and the list of dependent dialects. Based on this description, the `MyTilingPassBase` class was generated, containing `getArgument`, `getDescription`, `getDependentDialects` methods, along with `pass option` handling and the `registerMyTilingPass` function.

<details>
<summary>TableGen Description:</summary>

```mlir
def MyTilingPass : Pass<"my-tiling", "mlir::func::FuncOp"> {
  let summary = "Tile linalg operations with user-specified tile sizes";
  let description = [{
    Applies tiling transformation to linalg operations using TilingInterface.
    Takes tile sizes as pass option.
  }];

  let constructor = "mlir::createMyTilingPass()";
  let dependentDialects = ["mlir::linalg::LinalgDialect",
                            "mlir::scf::SCFDialect",
                            "mlir::memref::MemRefDialect", "mlir::tensor::TensorDialect"];

  let options = [
    ListOption<"tileSizes", "tile-sizes", "int64_t",
          "Tile sizes for each loop dimension">
  ];
}

```

</details>

The `MyTilingPass` class (see [MyTiling.cpp](/lib/Transforms/MyTilingPass.cpp)) inherits from the generated base class and overrides the `runOnOperation` method.

To apply the pattern-oriented approach, the TileUsingSCFPattern structure was implemented.

<details>
<summary>TileUsingSCFPattern Structure:</summary>

```c++
struct TileUsingSCFPattern : public OpInterfaceRewritePattern<TilingInterface> {
public:
  TileUsingSCFPattern(MLIRContext *ctx, scf::SCFTilingOptions options)
      : OpInterfaceRewritePattern<TilingInterface>(ctx),
        options(std::move(options)) {}

  LogicalResult matchAndRewrite(TilingInterface op,
                                PatternRewriter &rewriter) const override {
    if (op->hasAttr("tiled"))
      return failure();

    FailureOr<scf::SCFTilingResult> result =
        scf::tileUsingSCFForOp(rewriter, op, options);
    if (failed(result))
      return failure();

    for (Operation *newOp : result->tiledOps)
      newOp->setAttr("tiled", rewriter.getUnitAttr());

    rewriter.replaceOp(op, result->replacements);
    return success();
  }

private:
  scf::SCFTilingOptions options;
};

```

</details>

The pass itself registers this pattern and invokes `applyPatternsAndFoldGreedily` in `runOnOperation`.

<details>
<summary>runOnOperation Definition:</summary>

```c++
void MyTilingPass::runOnOperation() {
  func::FuncOp func = getOperation();
  MLIRContext *ctx = &getContext();
  Builder b(ctx);

  SmallVector<OpFoldResult> tileSizesOFR = llvm::to_vector(
      llvm::map_range(this->tileSizes, [&](int64_t v) -> OpFoldResult {
        return b.getIndexAttr(v);
      }));
  if (tileSizesOFR.empty())
    tileSizesOFR = {b.getIndexAttr(32), b.getIndexAttr(32), b.getIndexAttr(32)};

  scf::SCFTilingOptions tilingOptions;
  tilingOptions.setTileSizes(tileSizesOFR);

  RewritePatternSet patterns(ctx);
  patterns.add<TileUsingSCFPattern>(ctx, tilingOptions);

  if (failed(applyPatternsAndFoldGreedily(func, std::move(patterns))))
    signalPassFailure();
}
```

</details>

This approach ensures a single application of tiling to each operation, correctly handles cases where the operation is already inside a loop, and integrates well with other optimization passes.

To demonstrate the pass functionality, a separate tool `my-tiling-opt` was developed (see [my-tiling-opt.cpp](/tools/my-tiling-opt/my-tiling-opt.cpp)). In its `main` function, all `MLIR` dialects and the `pass` itself are registered, after which `MlirOptMain` is invoked. This allows the tool to function as a standard utility for transforming `MLIR` modules.

## Comparison of Approaches
During the development of the tiling pass, two approaches were utilized: an iterative pass over all instructions implementing the `TilingInterface`, and an approach based on traversing all patterns satisfying the `TilingInterface` via the `applyPatternsAndFoldGreedily` mechanism.

The iterative approach provided fine-grained control over the transformation process but required manual handling of dependencies between operations. In particular, it retained a potential vulnerability to invoking the transformation on an already deleted operation in the event of its replacement during tiling. This issue did not manifest in the test set; however, its presence creates a risk of errors when the optimization pipeline and test programs become more complex.

The pattern-oriented approach proved to be more high-level and convenient. The `GreedyPatternRewriter` infrastructure automatically managed the order of pattern application, tracked dependencies between operations, and correctly handled situations with deleted operations. The initial difficulty related to uncontrolled multiple applications of tiling to already processed operations was resolved by introducing a `tiled` attribute, which was set on operations after a successful transformation (as described in the [Tiling Pass Implementation](#tiling-pass-implementation) section).

In the final version of the pass, the pattern-oriented approach was selected, as it aligned with standard practices for developing passes in `MLIR`, ensured better compatibility with other optimizations, and demonstrated higher reliability and versatility.

## Results
The resulting tool, `my-tiling-opt`, when used with the `-my-tiling option`, executes the tiling pass on the input `mlir` program.

<details>
<summary>Transformation Example:</summary>

```mlir
module {
  func.func @matmul_64x64x64(%arg0: memref<64x64xf32>, %arg1: memref<64x64xf32>, %arg2: memref<64x64xf32>) {
    linalg.matmul ins(%arg0, %arg1 : memref<64x64xf32>, memref<64x64xf32>) outs(%arg2 : memref<64x64xf32>)
    return
  }
}

```

The input program performs a `linalg.matmul` operation on `memrefs` with dimensions divisible by the default tile sizes `{32,32,32}`. The output code is as follows:

```mlir
module {
  func.func @matmul_64x64x64(%arg0: memref<64x64xf32>, %arg1: memref<64x64xf32>, %arg2: memref<64x64xf32>) {
    %c0 = arith.constant 0 : index
    %c64 = arith.constant 64 : index
    %c32 = arith.constant 32 : index
    scf.for %arg3 = %c0 to %c64 step %c32 {
      scf.for %arg4 = %c0 to %c64 step %c32 {
        scf.for %arg5 = %c0 to %c64 step %c32 {
          %subview = memref.subview %arg0[%arg3, %arg5] [32, 32] [1, 1] : memref<64x64xf32> to memref<32x32xf32, strided<[64, 1], offset: ?>>
          %subview_0 = memref.subview %arg1[%arg5, %arg4] [32, 32] [1, 1] : memref<64x64xf32> to memref<32x32xf32, strided<[64, 1], offset: ?>>
          %subview_1 = memref.subview %arg2[%arg3, %arg4] [32, 32] [1, 1] : memref<64x64xf32> to memref<32x32xf32, strided<[64, 1], offset: ?>>
          linalg.matmul {tiled} ins(%subview, %subview_0 : memref<32x32xf32, strided<[64, 1], offset: ?>>, memref<32x32xf32, strided<[64, 1], offset: ?>>) outs(%subview_1 : memref<32x32xf32, strided<[64, 1], offset: ?>>)
        }
      }
    }
    return
  }
}

```

The output function body contains three nested `scf.for` loops iterating over blocks of size `32` along each dimension.

```
Outer loop over `%arg3` from `0` to `64` step `32` — iterates over row blocks of output matrix `C` and rows of matrix `A`.
Middle loop over `%arg4` from `0` to `64` step `32` — iterates over column blocks of output matrix `C` and columns of matrix `B`.
Inner loop over `%arg5` from `0` to `64` step `32` — iterates over the common dimension (`k`) for block multiplication.
```

Consequently, each `C` block of size `32×32` is computed as the sum of products of corresponding blocks from `A` and `B`. Within the innermost loop, three `subviews` are created using `memref.subview`:

```
`%subview` from `%arg0` (`A`): row block `[%arg3, %arg3+32)` and column block `[%arg5, %arg5+32)`.
`%subview_6` from `%arg1` (`B`): row block `[%arg5, %arg5+32)` and column block `[%arg4, %arg4+32)`.
`%subview_7` from `%arg2` (`C`): row block `[%arg3, %arg3+32)` and column block `[%arg4, %arg4+32)`.
```

All subviews have dimensions `32×32`, but their internal representation (`strided`) accounts for the original memory layout (row stride `64`, offset computed dynamically). This enables working with blocks without data copying. `linalg.matmul` multiplies the two submatrices (`A-block` and `B-block`) and accumulates the result into the corresponding `C` block. Thus, each block of the output matrix `C` is computed as the sum over all `k-blocks` of products.

## Conclusion
A tiling pass for the `Linalg` dialect within the `MLIR` infrastructure has been developed. The tool, implemented as a standalone optimization pass (`my-tiling-opt`), performs blocking of matrix multiplication operations with user-specified sizes for each dimension. The `TilingInterface` is used to identify suitable operations, while the transformation is implemented via `scf::tileUsingSCFForOp`, ensuring correct generation of nested `scf.for` loops and `subviews`.

The tool has been successfully tested on a wide range of configurations: square and rectangular matrices, static and dynamic dimensions, working with `memref` and `tensor`, as well as cases with existing loops and sequential multiplications.

Flexibility in configuration is ensured: `tile sizes` can be passed via a command-line option, with default values (`32×32×32`) used if none are explicitly provided.

The practical significance of the pass lies in automating a critical optimization for tensor computations – improving data locality and reducing memory accesses through block processing. This represents an important step towards building efficient compilers for deep learning and other computationally intensive tasks.

Potential directions for future development include: support for `multi-level` tiling (considering various levels of the memory hierarchy); and adding the capability for automatic `tile size` selection based on analysis of the target device architecture.

The developed tool can serve as a foundation for more complex transformations and demonstrates the effectiveness of leveraging the modern `MLIR` infrastructure for creating optimizing compilers.
